# Moviepedia Supabase Context & Protocols

> [!CAUTION]
> **CRITICAL WARNING FOR AGENTS:**
> When writing scripts to bulk-update this database, **NEVER** use variables defined outside your loop (e.g. `let role`) to store row-specific data.
> **ALWAYS** ensure variables like `role`, `cast`, or `id` are scoped strictly *inside* the loop (using `const` or `.map()`).
> Failing to do this causes "Data Pollution" where data from Movie A (e.g. "Veer") is written into Movie B, C, D...
> **Verify distinctness** before committing bulk updates.

## 9. ID Naming Conventions (CRITICAL)
We enforce strict prefixes for IDs to avoid duplicates and legacy messes.
*   **Movies**: `tmdb-12345` (Prefix `tmdb-`)
*   **People (Cast/Crew)**: `tmdb-person-12345` (Prefix `tmdb-person-`)
    *   ❌ **NEVER USE**: `cast-12345` (Legacy/Incorrect)
    *   ❌ **NEVER USE**: `director-Name` (Unless strictly for searching by string match before ID resolution)

## 10. Table Schemas

### `cast` Table
Used for storing cached actor/director details.
*   **Primary Key/Constraint**: `tmdb_id` (Text) - *Must match `tmdb-person-XXXX` format.*
*   **Columns**:
    *   `tmdb_id` (Unique Identifier)
    *   `name` (Text) - *Note: NOT Unique. Constraint `cast_name_key` should be DROPPED.*
    *   `biography` (Text) - *Important: Store `""` (empty string) for missing bios, NOT `null`.*
    *   `image_url` (Text)
    *   `birthday` (Date)
    *   `deathday` (Date)
    *   `place_of_birth` (Text)
    *   `known_for_department` (Text)
*   **Behavior**: This table is a **Read-Through Cache**.
    1.  Frontend checks DB.
    2.  If missing/bad data, fetch TMDB.
    3.  Upsert result to DB.

### `movies` Table
*   **Key Column**: `data` (JSONB)
    *   Stores the entire movie object including `slug`, `cast`, `crew`, `streamingLinks`, etc.
    *   **CRITICAL**: The `slug` is **NOT** a top-level column. It is located at `data->slug`.
    *   To get the slug, you must select `data` and parse it (e.g., `data->slug` or in JS `movie.data.slug`).
*   **Foreign Keys**:
    *   There is NO database-level Foreign Key between `articles` and `movies`.
    *   Joins (e.g., `select('*, movie:movies(slug)')`) **WILL FAIL**.
    *   **Pattern**: Fetch the article first, then use `related_movie_id` to manually fetch the movie's `data` column in a separate query.

## 11. RLS Policies (Security)
The frontend accesses Supabase as an `anon` user.
*   **Read**: Publicly enabled.
*   **Update/Insert**: MUST be enabled for `public`/`anon` on the `cast` table for the caching mechanism to work.
    *   *SQL*: `create policy "Enable update for all users" on "public"."cast" ...`

## 12. Common Pitfalls & Rules
1.  **Do NOT assume Name uniqueness**: People share names. Always rely on `tmdb_id` for updates.
2.  **Empty Bio Logic**: When caching a person with no bio from TMDB, save it as an **empty string (`""`)**. This acts as a flag that "we checked and found nothing," preventing infinite re-fetch loops. Saving `null` causes the app to think the data is "missing" and try again.
3.  **Local vs Global State**: The `PersonPage` often fetches its own "Local Movies" to avoid race conditions with the global context. trust the local fetch logic.
4.  **Skeleton Loading**: The `PersonPage` uses a custom `PersonSkeleton` component, not a generic spinner.
